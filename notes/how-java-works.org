*** JVM (Java Virtual Machine)

دي الEngine نفسه اللي بيشغل الكود وبيحول الbyte code ل machine code

مع ان الbyte code مش Platform Independent الا ان الـJVM هو Platform Independent

ليه ؟ لان لكل نظام تشغيل زي windows, linux, mac, etc في نسخة JVM خاصة ليه بتحول الbyte code لmachine code يشتغل مخصوص علي النظام ده لكن الbyte code نفسه واحد

عشان كده Java هي "Write once, Run everywhere"

**محتوايتها:**

| Class Loader       | بيحمل الclass files علي الmemory                |
|--------------------+-----------------------------------------------|
| Byte code verifier | بيكشف علي الbyte code                          |
| Exectution Engine  | ده اللي بيتحكم في الInterpreter و الJIT            |
| Garbage Collector  | بيدير الميموري وبيمسح اللي مش محتاجه                  |
| Runtime Data Areas | الميموري: Heap, Stack, Method Area, PC Register |

### ليه الـ Linking بيحصل "Dynamic" في جافا؟

بعكس لغة C++ اللي الـ Linker بتاعها بيخلص كل حاجة قبل ما البرنامج يشتغل في link time (Static Linking)، جافا بتعمل **Dynamic Linking**.

**يعني ايه؟** يعني لو عندك كلاس `A` بيستخدم كلاس `B`.. الـ JVM مش هيعمل Resolve لعنوان كلاس `B` إلا لما البرنامج يحتاج يستخدمه فعلاً "On the fly". ده اللي بيخليك تقدر تضيف Libraries جديدة أو تعمل Update لملفات `.class` والبرنامج شغال من غير ما تعيد بناء البرنامج كله.

*** المرحلة الرابعة :

**5- Execution Engine**

**الJVM بتستخدم نظام هجين مكون من interpreter و JIT compiler عشان تحقق توازن بين السرعة في التشغيل وكفائة الاداء**

1. **الـ Interpreter (البداية السريعة)**

الـ Interpreter وظيفته إنه يقرأ الـ Bytecode ويترجمه للغة الآلة **سطر بسطر** وينفذه فوراً.

- **الميزة:** البرنامج بيبدأ يشتغل "لحظياً". مش محتاج يستنى وقت طويل عشان يترجم الكود كله قبل ما يبدأ.
- **العيب:** بطيء في التنفيذ المتكرر. لو عندك `loop` بتلف مليون مرة، الـ Interpreter هيترجم نفس السطر مليون مرة، وده ضياع كبير للوقت والمجهود.

----------------------------------------------------------------------------------------

1. **الـ JIT Compiler (الأداء العالي)**

الـ **Just-In-Time (JIT)** هو "الخبير" اللي بيراقب البرنامج وهو شغال.

- **وظيفته:** بيدور على الـ **"Hot Spots"**، وهي أجزاء الكود اللي بتتنفذ كتير (زي الـ Loops أو الدوال المتكررة).
- **الميزة:** لما بيلاقي "Hot Spot"، بياخد الكود ده ويترجمه بالكامل لـ **Native Machine Code** (لغة الآلة الأصلية) وبيعمل له Optimized (تحسين) فائق، وبيخزنه في الـ **Code Cache**.
- **النتيجة:** المرة الجاية لما البرنامج يحتاج ينفذ الكود ده، مش هيروح للـ Interpreter، هينفذ النسخة الجاهزة والسريعة جداً مباشرة من الميموري

----------------------------------------------------------------------------------------

**النظام الهجين بيعمل الآتي:**

1. الـ **Interpreter** بيبدأ يشغل البرنامج فوراً عشان المستخدم ميحسش بتأخير.
2. أثناء ما البرنامج شغال، الـ **JIT** بيحلل الكود في الخلفية (Background).
3. بمجرد ما الـ JIT يجهز نسخة سريعة من الأجزاء المهمة، البرنامج "بيبدل" سرعته من سرعة الـ Interpreter لسرعة الـ Native Code.

----------------------------------------------------------------------------------------

**ليه محتاجين "الاثنين" مع بعض؟**

تخيل لو كان عندنا واحد منهم بس:

- **لو Interpreter بس:** البرنامج هيبدأ بسرعة، بس هيفضل "سحلفاة" طول ما هو شغال لأن الترجمة سطر بسطر مرهقة للبروسيسور.
- **لو JIT بس:** هتضطر تستنى "سنة" في الأول (Start-up time) عشان الـ JIT يترجم ويحسن كل الكود قبل ما البرنامج يفتح، وده شيء محبط جداً للمستخدم

تخيل إنك عندك Loop واحدة بتلف **مليار مرة** جوه دالة اتنادت **مرة واحدة بس**. لو الـ JIT استنى الدالة تخلص

عشان يحسنها للمرة الجاية، يبقى ملوش لازمة لأن المليار لفة هيكونوا خلصوا ببطء!

1. **الـ OSR (On-Stack Replacement) - "التبديل على الهواء"اللي بيحصل هنا:**
2. الـ JVM عنده عداد للـ Back-edges (يعني كم مرة الـ Loop رجعت لبدايتها).
3. لما الـ Loop "تسخن" جداً وهي لسه شغالة، الـ JIT بيترجم كود الـ Loop دي فوراً في الخلفية.
4. الـ JVM بيقوم بعملية انتحارية تقنياً اسمها **On-Stack Replacement**: بيستبدل كود الـ Loop اللي شغال حالياً (في الـ Stack) بالنسخة السريعة المترجمة **بدون ما يوقف البرنامج**.
5. البرنامج بيكمل باقي اللفات (مثلاً من اللفة رقم 50,000 لحد المليار) بالسرعة القصوى.
